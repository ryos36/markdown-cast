# markdown-cast

Marp の Markdown スライドを起点に、**字幕（SRT）・字幕焼き込み動画・ナレーション音声つき動画**を作るパイプライン。
スライドに埋め込んだ発話ノート（原稿）を、形態素解析 → 文節 → 字幕 / TTS 音声に変換し、
**音声の長さに合わせて字幕とフレームのタイミングを自動で調整する**のが中心の仕事。

ビルドは `ninja`、変換ツールは Common Lisp（Roswell / `ros`）と awk、
音声・動画は Azure TTS / `sox` / `ffmpeg` / `gstreamer` / Marp CLI を使う。

> このリポジトリは、別プロジェクト（子ども向けイベントの企画スライド動画化）で育てたパイプラインを
> 切り出して一般化したもの。まずは元と同じ構成を再現し、そこから固有部分（日本語・Azure 依存）を
> 段階的に外していく方針。

---

## 何ができるか

入力は 1 枚の Marp Markdown（`deck.md`）。発話ノートを HTML コメント `<!-- ... -->` で各スライドに書いておく。
そこから次を作る。

| 出力 | 中身 |
|---|---|
| `deck.pdf` | Marp の PDF |
| スライド PNG | 1 スライド 1 枚 |
| `subs.srt` | 字幕 |
| `*-with-caption.mp4` | 字幕を焼き込んだ動画（音声なし） |
| `*-with-caption-audio.mp4` | 字幕＋ナレーション音声つき動画 |
| `*-final-caption-audio.mp4` | 速度調整でフレームを足し、通し番号を振り直した最終版 |

`ninja wav`（Azure TTS で課金）以外は Azure 不要。タイミング等は `build.ninja` 冒頭の変数で調整する。

---

## 全体の流れ

```
deck.md
 ├─ marp --pdf                                  → PDF
 ├─ marp --images png                           → スライド PNG（orig-png）
 └─ note2ss.awk  発話ノートを S 式リストに        → note.ss
     └─ mecab 形態素解析 → 文節化（複合語は辞書で1語に）
        ├─ → caption（字幕の単位）化
        │    ├─ caption2srt    → SRT 字幕
        │    └─ caption2pnglist → フレーム複製（ハードリンク）→ 字幕焼き込み動画
        └─ → 文ごとの読み上げ単位（bunsetu2tts）
             └─ tts2wav（Azure）→ 文ごとの wav
                └─ tts2wavlist  音声の実測尺からタイミングを計算
                   └─ sox tempo で速度調整 → ffmpeg で動画に音声を重ねる
```

## 中心の考え方：「文節 = 1 フレーム = 一定時間」という共通の時間軸

音声・字幕・画像をバラバラに作ると同期が合わない。このパイプラインは、すべてを **1 つの番号体系**に乗せる。

- **文節**を最小単位とする（形態素解析の語をいくつかまとめたもの）。
- 字幕は文節をいくつか詰めた **caption**。**1 caption = 1 フレーム = `capms` ミリ秒**（既定 800ms）。
- フレームには出現順に通し番号がつく。SRT のエントリ番号も、スライド PNG を複製した動画フレームも、
  音声を差し込む時刻も、**全部この同じ番号で決まる**。
- フレーム `i` の時間区間は `[(i-1)*capms, i*capms)`。動画の framerate も `1000/capms` fps にして一致させる。

この「番号 = 時刻」の取り決めが、後段の音声合わせをシンプルにしている。

## 音声とテキストを合わせる工夫

ここがこのプロジェクトの肝。順に説明する。

### 1. 原稿はスライドに同梱する

発話ノート（読み上げ原稿）を Marp の HTML コメント `<!-- ... -->` に書く。スライドと原稿が 1 ファイルに収まり、
ズレようがない。`note2ss.awk` が各スライドのコメントを連結し、`("1枚目の原稿" "2枚目の原稿" ...)` という
1 個の S 式リストにする（原稿の無いスライドは飛ばす）。

### 2. 形態素解析 → 文節化、そして「句読点で文を切る」

`note2word`（mecab）で各文を `((単語 . 品詞) ...)` にし、`word2bunsetu` で文節にまとめる。区切りの規則：

- 単語の表層を順に連結していく。
- **助詞・副詞**を足したらそこで 1 文節を区切る（助詞・副詞は手前の文節にくっつく）。
- **句読点「。」「、」が来たら、文節を区切り、かつ文（外側のリスト）も区切る**。
  句読点は捨てず直前の文節にくっつける（字幕に句読点が残り、読点の位置で間が入る）。
- **全角空白「　」**は手動の文節区切り（空白自体は捨てる）。mecab の分け方が気に入らないとき、原稿側で
  「　」を打って区切りを指定できる（半角空白は mecab が捨てるので使わない）。
- 英数字 ASCII 語が連続するときだけ単語間に半角スペースを入れる（`top of the world` は空白あり、
  `micro:bit` は記号混じりなので空白なし、`ryosさん` は日本語混じりなので空白なし）。

ポイントは、**句点「。」が「文 = TTS の読み上げ単位」の切れ目になり、句読点が字幕にもそのまま残る**こと。
人が読む区切りと、機械が読み上げる区切りと、字幕の見た目を、句読点ひとつで一致させている。

複合語（例：「micro:bit」「Switch Science」）が mecab でバラバラに割れる問題は、**複合語辞書**
（`mecab-private.dict.ss`、語リスト）を `word2word` が見て元の 1 語に畳み直すことで防ぐ。

### 3. 字幕の組み方：文節は割らず、N 文字で折り返す

`bunsetu2caption` が各文の文節を前から累積し、合計が N 文字（既定 10）を超えたら窓をリセットして次の
caption を始める。**文節そのものは絶対に割らない**（1 文節が N 超でも単独 caption）。
1 文節につき caption が 1 つ進むので、「文の開始 caption 番号 = それまでの文節数の合計 + 1」で求まる。
この番号が、後段で音声を差し込む位置になる。

### 4. TTS は「文」単位でまとめて読み上げる

字幕は文節ごとに切り替わるが、**音声を文節ごとにぶつ切りで合成すると不自然**。そこで `bunsetu2tts` が
文（句点までの文節列）を 1 つの読み上げ単位にまとめ、`((開始フレーム番号 文) ...)` を作る。
TTS は文をまるごと読み、その音声が開始フレームから始まる、という対応にする。

### 5. 読み辞書は「音声側だけ」に当てる

`pronunciation.dict.ss` は `("micro:bit" . "マイクロビット")` のような **見出し . 読み** の対。
`tts2wav` はこれを **SSML（読み上げ）側だけ**に適用する。**字幕は元の表記のまま、音声だけ読みを変える**ので、
画面には `micro:bit` と出しつつ「マイクロビット」と読ませられる。

### 6. SSML の間（break）と「末尾の無音を焼かない」工夫

`tts2wav` は文節を連結し、**文節間に `break_ms` の小さな間**、文末に `end_ms` の間を入れた SSML を合成する。
`end_ms` は **0 推奨**。理由は、wav の末尾に無音を焼き込むと、後段で全体を `sox tempo` で速めたときに
その無音まで一緒に速まり、さらに mux 側の `tail_ms`（末尾余白）と二重になるため。
**文末の間は wav に焼かず、合成時の `tail_ms` で持たせる**。

### 7. 音声の実測尺に合わせてフレーム数を可変にする（factor）

ここが核心。各文の wav の**実際の再生時間 `ms`**（`sox --i -D` で実測。逆算しない）と、
その文に割り当てられたフレーム数 `B`（= 次の文の開始番号 − この文の開始番号）から、詰め具合を決める。

```
avail  = B*capms - head_ms - tail_ms      （その文が使える実時間）
factor = max(1.0, ms / avail)             （音声が長すぎるぶんだけ速める係数）
```

`factor` 倍速で `sox tempo` をかけ、音声を割当時間に収める。**速める（縮める）だけで、遅くはしない**
（`max 1.0`）。短い文はそのまま、長い文だけ少し速くなる。

### 8. 速すぎ防止：フレームを足して、通し番号を振り直す

`factor` が大きすぎると早口で聞き取れない。そこで **`threshold`（既定 1.2）を超える文は、フレームを `n` 枚
足して**（割当を `(B+n)*capms` に伸ばして）`ms/((B+n)*capms) < threshold` になる最小の `n` を選び、
速度を `threshold` 未満に抑える。足したフレームは**末尾 caption（字幕）を複製**して埋めるので、字幕表示も
そのぶん伸び、音声と一緒にゆっくりになる。

フレームを足すと**後続の全フレームの通し番号が後ろにずれる**（足した枚数の累計 `cumsum`）。
そこで `final` 系のツールが、画像（`ss2finalpng`）・字幕（`ss2finalsrt`）・音声（`final_audio_mux`）を
**新しい通し番号に合わせて作り直し**、再同期する。音声の無い後半フレームも、累積ずれぶんずらして 1:1 で並べ直す。

これらのタイミング計算（`B` / `ms` / `factor` / `n` / new-rate / 通し番号）を 1 つの S 式 `wavlist.ss` に
まとめておき、画像・字幕・音声の 3 つが同じ地図を見て動く。

### 9. 頭出しと合成

各文の音声を置く時刻（遅延）は `delay = head_ms + (開始フレーム番号 − 1) * capms`。
全文の wav を `ffmpeg` の `adelay` でそれぞれ遅らせ、`amix` で 1 本に重ねて動画に乗せる。
フレーム頭ぴったりに各文が鳴り出す。

### 10. wav のキャッシュで Azure 課金を抑える

`tts2wav` は **読み適用後の文節列＋音声設定**をキーに `tts-info.ss` を持ち、
**同じ読み・同じ設定の wav は再生成しない（Azure を呼ばない）**。設定が変わった既存 wav は番号つきで
`backup-wav/` へ退避してから作り直す。`--bootstrap` で、手元にある wav から info を起こして再利用可能に
する初期化もできる。Azure が途中までしか作れていない場合は、**連続して揃っている前半だけ**を出力して打ち切る。

## 必要なもの

`ros`（Roswell）/ `mecab` / `gst-launch-1.0` / `ffmpeg` / `sox` / `npx`（Marp CLI）。
TTS には Azure Speech のキー（`key.ninja` の `$key`。コミットしないこと）。

## 主なターゲット

```
ninja pdf                     # PDF
ninja png                     # スライド PNG
ninja srt                     # 字幕 SRT
ninja with-caption.mp4        # 字幕焼き込み動画
ninja wav                     # ★Azure TTS で各文を wav 化（課金注意）
ninja with-caption-audio.mp4  # 音声つき動画
ninja final-caption-audio.mp4 # 速度調整・通し番号振り直しの最終版
```

`build.ninja` 冒頭の変数：`capms`（1 フレーム長）, `head_ms`/`tail_ms`（前後余白）,
`break_ms`/`end_ms`（SSML の間）, `threshold`（速すぎ防止の閾値）, `mecab_private_dic`（複合語辞書）など。

## ツール一覧

| ファイル | 種別 | 役割 |
|---|---|---|
| `note2ss.awk` | awk | 発話ノート（`<!-- -->`）を 1 個の S 式リストにする |
| `note2word.ros` | mecab | テキストを `((単語 . 品詞) …)` にする |
| `word2word.ros` | filter | 複合語辞書で mecab の分割を 1 語に畳む |
| `word2bunsetu.ros` | filter | 助詞・副詞・句読点で区切って文節にまとめる |
| `bunsetu2caption.ros` | filter | 文を字幕（caption）の列にする（N 文字で折り返し） |
| `mapcar.ros` | 単体 | S 式の各要素に filter チェーンを mapcar する汎用ツール |
| `caption2srt.ros` | 単体 | caption を平坦化して固定長の SRT にする |
| `caption2pnglist.ros` | 単体 | caption からフレーム複製の元/先名の対を作る |
| `bunsetu2tts.ros` | 単体 | 文節構造から「文ごと」の読み上げ単位を作る |
| `tts2wav.ros` | Azure | 各文を Azure TTS で wav 化（読み辞書・SSML・キャッシュ） |
| `tts2wavlist.ros` | 単体 | 音声の実測尺から factor / n / 通し番号を計算し `wavlist.ss` を作る |
| `ss2finalpng.ros` | 単体 | 通し番号でフレームを並べ直す（ハードリンク） |
| `ss2finalsrt.ros` | 単体 | 通し番号に合わせて SRT を再タイミング |
| `mux_audio.sh` / `final_audio_mux.sh` | ffmpeg | wav を時刻配置して動画に重ねる |

補助：`ss2wav.ros`（単発 TTS）, `sort.ros` / `uniq.ros` / `nth.ros` / `pretty.ros` / `parsenum.ros` 等。

## いまの固有部分（一般化の対象）

- **日本語前提**：mecab の形態素解析・文節化・句読点規則は日本語向け。他言語では segmentation 部を差し替える必要がある。
- **Azure TTS 前提**：`tts2wav` が Azure の SSML / issueToken に直結。バックエンドを抽象化できると汎用になる。
- **Common Lisp（ros）依存**：変換ツールが Roswell + SBCL 前提。利用のハードルになるので、いずれ移植を検討。

これらを差し替え可能にしていくのが、このリポジトリの今後の作業。
