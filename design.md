# 設計思想

## 中心の考え方：「文節 = 1 フレーム = 一定時間」という共通の時間軸

音声・字幕・画像をバラバラに作ると同期が合わない。このツールは、すべてを **1 つの番号体系**に乗せる。

- **文節**を最小単位とする（形態素解析の語をいくつかまとめたもの）。
- 字幕は文節をいくつか詰めた **caption**。**1 caption = 1 フレーム = `capms` ミリ秒**（既定 800ms）。
- フレームには出現順に通し番号がつく。SRT のエントリ番号も、スライド PNG を複製した動画フレームも、
  音声を差し込む時刻も、**全部この同じ番号で決まる**。
- フレーム `i` の時間区間は `[(i-1)*capms, i*capms)`。動画の framerate も `1000/capms` fps にして一致させる。

この「番号 = 時刻」の取り決めが、後段の音声合わせをシンプルにしている。

## 音声とテキストを合わせる工夫

ここがこのプロジェクトの肝。順に説明する。

### 1. 原稿はスライドに同梱する

発話ノート（読み上げ原稿）を Marp の HTML コメント `<!-- ... -->` に書く。スライドと原稿が 1 ファイルに収まり、
ズレようがない。`note2ss.awk` が各スライドのコメントを連結し、`("1枚目の原稿" "2枚目の原稿" ...)` という
1 個の S 式リストにする（原稿の無いスライドは飛ばす）。

### 2. 形態素解析・文節化、そして「句読点で文を切る」

`note2word`（mecab）で各文を `((単語 . 品詞) ...)` にし、`word2bunsetu` で文節にまとめる。区切りの規則：

- 単語の表層を順に連結していく。
- **品詞が「助詞」または「副詞」**の語を足したらそこで 1 文節を区切る。
  助詞は直前の文節にくっつく。副詞は独立して文節を閉じる（直前にはくっつかない）。
- **句読点「。」「、」**は品詞ではなく**表層（文字）で判断する**。
  mecab が何の品詞として返しても、表層が `。` か `、` なら文節と文（外側のリスト）を区切る。
  区切りに使った句読点は、既定では直前の文節にくっつける（字幕に残る）。
  `--strip-punctuation` を指定すると文節テキストから落とす（字幕・音声の両方に効く）。
- **全角空白「　」**は手動の文節区切り（空白自体は捨てる）。mecab の分け方が気に入らないとき、原稿側で
  「　」を打って区切りを指定できる（半角空白は mecab が捨てるので使わない）。
- 英数字 ASCII 語が連続するときだけ単語間に半角スペースを入れる（`top of the world` は空白あり、
  `micro:bit` は記号混じりなので空白なし、`ryosさん` は日本語混じりなので空白なし）。

ポイントは、**句点「。」が「文 = TTS の読み上げ単位」の切れ目になる**こと。
人が読む区切りと、機械が読み上げる区切りを、句読点ひとつで一致させている。
句読点を字幕・音声テキストに残すか落とすかは `--strip-punctuation` で選べる。

複合語（例：「micro:bit」「Switch Science」）が mecab でバラバラに割れる問題は、**複合語辞書**
（`mecab-private.dict.ss`、語リスト）を `word2word` が見て元の 1 語に畳み直すことで防ぐ。
畳んだ語の品詞は常に `"単語"` になる。助詞・副詞でも句読点でもないので、文節を閉じず
そのまま連結対象になる（複合語が意図せず文節を切らないための設計）。

### 3. 字幕の組み方：文節は割らず、N 文字で折り返す

`bunsetu2caption` が各文の文節を前から累積し、合計が N 文字（既定 10）を超えたら窓をリセットして次の
caption を始める。**文節そのものは絶対に割らない**（1 文節が N 超でも単独 caption）。
1 文節につき caption が 1 つ進むので、「文の開始 caption 番号 = それまでの文節数の合計 + 1」で求まる。
この番号が、後段で音声を差し込む位置になる。

### 4. TTS は「文」単位でまとめて読み上げる

字幕は文節ごとに切り替わるが、**音声を文節ごとにぶつ切りで合成すると不自然**。そこで `bunsetu2tts` が
文（句点までの文節列）を 1 つの読み上げ単位にまとめ、`((開始フレーム番号 文) ...)` を作る。
TTS は文をまるごと読み、その音声が開始フレームから始まる、という対応にする。

### 5. 読み辞書は「音声側だけ」に当てる

`pronunciation.dict.ss` は `("micro:bit" . "マイクロビット")` のような **見出し . 読み** の対。
`tts2wav` はこれを **SSML（読み上げ）側だけ**に適用する。**字幕は元の表記のまま、音声だけ読みを変える**ので、
画面には `micro:bit` と出しつつ「マイクロビット」と読ませられる。

### 6. SSML の間（break）と「末尾の無音を焼かない」工夫

`tts2wav` は 1 つの文（句点で区切られた読み上げ単位）の文節列を連結し、SSML を作る。
**同一文の文節と文節の間**に `<break time="break_ms ms"/>` タグを挿入し、**文末**に `<break time="end_ms ms"/>` を付けて合成する。

```xml
<speak ...><voice ...><prosody ...>
  わたしは<break time="120ms"/>良い<break time="120ms"/>ですね<break time="0ms"/>
</prosody></voice></speak>
```

`break_ms` は**文間（文と文の間）には入らない**。句点「。」で区切られた別の文はそれぞれ独立した SSML・別の wav として合成される。

`end_ms` は **0 推奨**。理由は、wav の末尾に無音を焼き込むと、後段で全体を `sox tempo` で速めたときに
その無音まで一緒に速まり、さらに mux 側の `tail_ms`（末尾余白）と二重になるため。
**文末の間は wav に焼かず、合成時の `tail_ms` で持たせる**。

### 7. 音声の実測尺に合わせてフレーム数を可変にする（factor）

ここが核心。各文の wav の**実際の再生時間 `ms`**（`sox --i -D` で実測。逆算しない）と、
その文に割り当てられたフレーム数 `B`（= 次の文の開始番号 − この文の開始番号）から、詰め具合を決める。

```
avail  = B*capms - head_ms - tail_ms      （その文が使える実時間）
factor = max(1.0, ms / avail)             （音声が長すぎるぶんだけ速める係数）
```

`factor` 倍速で `sox tempo` をかけ、音声を割当時間に収める。**速める（縮める）だけで、遅くはしない**
（`max 1.0`）。短い文はそのまま、長い文だけ少し速くなる。

### 8. 速すぎ防止：フレームを足して、通し番号を振り直す

`factor` が大きすぎると早口で聞き取れない。そこで **`threshold`（既定 1.3）を超える文は、フレームを `n` 枚
足して**（割当を `(B+n)*capms` に伸ばして）`ms/((B+n)*capms) < threshold` になる最小の `n` を選ぶ。

フレームを追加した後の実際の速度係数（`new-rate`）は：

```
new-rate = ms / ((B+n) * capms)
```

`avail`（head/tail を引いた正味時間）ではなく **`(B+n)*capms` の全コマ時間で割る**点に注意。
`new-rate` は threshold 未満、かつ 1.0 以上になる。

音声の最終処理（`final_audio_mux.sh`）では `new-rate > 1.0` のとき `sox tempo new-rate` を適用し、
`new-rate <= 1.0`（音声が割当より短い場合）はそのままコピーする。

足したフレームは**末尾 caption（字幕）を複製**して埋める。「末尾 caption」はその文の最後の caption であり、
複数文節が 1 caption にまとまっている場合はまとめたテキストがそのまま繰り返し表示される。

**数値例**（既定値 capms=800, head_ms=50, tail_ms=10, threshold=1.3）:

ある文が 2 caption にまたがり（B=2）、wav の実測が 2800ms だった場合:

```
avail  = 2*800 - 50 - 10 = 1540ms
factor = max(1.0, 2800/1540) = 1.818   （> 1.3 なのでフレームを追加）

n=1: 2800 / ((2+1)*800) = 2800/2400 = 1.167  （< 1.3 で成立。これが最小の n）

new-rate = 1.167（sox tempo 1.167 を適用して 1.167 倍速に）
この文は 2 フレームから 3 フレームに伸び、3 枚目は末尾 caption を複製
```

cumsum と running の連動（この文を文1、次の文を文2 とする場合）:

```
文1: A=1, B=2, n=1  →  running = 1 + 0(cumsum) = 1
                         処理後 cumsum = 0 + 1 = 1
文2: A=3            →  running = 3 + 1(cumsum) = 4
                         文2 の音声差込遅延 = 50 + (4-1)*800 = 2450ms
```

文1 でフレームを 1 枚追加したことで、文2 以降の通し番号が 1 つ後ろにずれ、
音声の差込位置も自動的にずれる。累計ずれ量を `cumsum` として管理し、
各文の「final での通し番号（`running`）= 元の A + cumsum」が wavlist.ss に記録される。

`final` 系のツールは `running` を拠り所に画像（`ss2finalpng`）・字幕（`ss2finalsrt`）・音声（`final_audio_mux`）を
**新しい通し番号に合わせて作り直し**、再同期する。wavlist.ss に載らない音声なし後半フレームも、
累積ずれぶんだけずらして 1:1 で並べ直す。

これらのタイミング計算（`B` / `ms` / `factor` / `n` / `new-rate` / `running`）を 1 つの S 式 `wavlist.ss` に
まとめておき、画像・字幕・音声の 3 つが同じ地図を見て動く。

### 9. 頭出しと合成

各文の音声を置く遅延は、フレーム追加後の通し番号 `running` を使って計算する：

```
delay = head_ms + (running - 1) * capms
```

`running` は cumsum 加算後の番号なので、フレームを追加した文より後の文は自動的に後ろにずれた位置に置かれる。
全文の wav を `ffmpeg` の `adelay` でそれぞれ遅らせ、`amix` で 1 本に重ねて動画に乗せる。
各文の音声はそのフレームの先頭から `head_ms` 後に鳴り出す。

### 10. wav のキャッシュと Azure 課金のガード

`tts2wav` は **読み適用後の文節列＋音声設定**をキーに `tts-info.ss` を持ち、
**同じ読み・同じ設定の wav は再生成しない（Azure を呼ばない）**。設定が変わった既存 wav は番号つきで
`backup-wav/` へ退避してから作り直す。`--bootstrap` で、手元にある wav から info を起こして再利用可能に
する初期化もできる。Azure が途中までしか作れていない場合は、**連続して揃っている前半だけ**を出力して打ち切る。

課金を抑えるガードは主に 3 つ:

1. **`--dry-run`**: Azure を呼ばず、再生時間を「文節数 × 固定値（既定 200ms）」で見積もる。
   wav は作らない。`key.ninja` の初期値は `tts_opt = --dry-run` なので、キーを設定せずに
   パイプライン全体の流れを確認できる。
2. **`limit`**: 最初の N ページだけ音声化して止まる。`key.ninja` の `limit = 6` で設定。
   全ページ音声化するときは `limit = 0`。
3. **キャッシュ**: 同じテキスト・設定の wav があれば Azure を呼ばず再利用する（上記）。

なお `--dry-run` のまま（wav が 0 本）でも `video-audio` は作られる。
音声の代わりにプリセット音（440Hz 0.3 秒）が先頭に一度鳴るだけで、実質ほぼ無音の動画になる。

### 12. 動作の細かい仕様

**文節区切りの特殊ルール（word2bunsetu）**

- **全角空白「　」** を原稿に書くと手動で文節を区切れる。mecab の分け方が気に入らないときに使う。半角空白は mecab が捨てるので効かない。
- **助詞が文節閉じ直後に来た場合**（buf が空のとき）、新しい空文節を作らず直前の文節にくっつける。
- **ASCII 英数字が連続するときだけ**単語間に半角スペースを自動挿入する（例: `top of the world`）。記号や日本語が混じる語（`micro:bit`、`ryosさん`）には入らない。
- **句読点「。」「、」**は品詞でなく表層（文字）で判断する。mecab が何の品詞として返しても、`。` か `、` なら文節と文を区切る。

**読み辞書は音声側だけに効く（tts2wav）**

`pronunciation.dict.ss` の置換（例: `("micro:bit" . "マイクロビット")`）は SSML の読み上げ側だけに適用される。字幕は元の表記のまま表示される。

書き方は dot 対（`"見出し" . "読み"`）。`("micro:bit" "マイクロビット")` のようにドットなしで書くとエラーになる。

**空文字列は未設定扱い（tts2wav）**

ninja 変数が未設定のまま空文字列として渡された場合、`tts2wav` は空文字列を nil 扱いにする。

**wav の処理は先頭から順に、欠けたところで打ち切り（tts2wavlist）**

A 番号順に wav を処理し、最初に wav が存在しない番号に当たった時点で打ち切る。それ以降の出力は作らない。途中まで音声化した状態でビルドを動かすときの挙動。
